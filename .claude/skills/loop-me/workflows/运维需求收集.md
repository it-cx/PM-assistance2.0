# 运维需求随手收集

将口头传达的运维优化需求即时收录到需求池，防止遗漏，并自动关联去重。

## Trigger
**事件触发**——用户在 Claude Code 中发送模糊需求记录时自动触发。识别规则：
- 用户说"运维需求：xxx"、"记录一个需求"、"收到一个需求"等信号词
- 内容为简短的功能描述或优化要求（通常一句话）

## 流程

### 入库阶段
1. 用户粘贴一句话需求描述
2. AI 做三件事：
   - **补全字段**：日期自动填当天，若用户没说来源则追问"谁提的？"，若用户没说项目则根据上下文推测（云控/车联网/通用）
   - **去重检查**：对比 `需求池.md` 中已有条目，若疑似重复则提醒用户"这可能和 #N 重复，确认一下？"
   - **优先级猜测**：根据需求性质给出建议优先级（P0 故障 / P1 体验 / P2 优化 / P3 锦上添花），供用户确认
3. 用户确认后，追加到 `需求池.md`

### 汇总阶段
每月排期时，用户说"整理待排期需求"，AI 读取 `需求池.md` 中状态为"待评审"的条目，按优先级排序输出，供用户挑选哪些进当月迭代。

## 需求条目格式
```
| # | 日期 | 来源 | 项目 | 需求简述 | 优先级 | 状态 |
```

### 状态流转
```
待评审 → 已排期（进了某月迭代）
待评审 → 已拒绝（明确不做，附原因）
已排期 → 已完成（上线后）
```

### 示例
```
| 1 | 2026-07-07 | 运维-张工 | 云控 | 机器人状态监控页加载太慢，客户投诉 | P1 | 待评审 |
| 2 | 2026-07-07 | 客户-XX公司 | 车联网 | 车辆轨迹回放需要支持多车同时播放 | P2 | 待评审 |
```

## 边界
- 用户不说"运维需求"等信号词而只是在聊天中随口提到，AI 不自动记录——只在明确的入库意图下触发
- AI 不替用户决定优先级，仅建议，最终以用户确认为准
- 拒绝的需求保留在池中（标为"已拒绝+原因"），防止重复提
