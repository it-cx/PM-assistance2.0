# 第一部分：产品定义

## 1. 一句话定位

一个部署在公司 Agent 平台上的 **车辆工程分析助手**，让制造厂的工程师用自然语言对话的方式排查车辆问题、关联非结构化信息、发现未知的关联——而不是再看一个更细的看板。

## 2. 用户角色

| 角色 | 典型场景 | 核心诉求 |
|------|---------|---------|
| **质量工程师** | 盯着已售车辆的异常趋势，找批次问题 | "这批车的同一个部件是不是都有问题？" |
| **产品改进工程师** | 分析实际运营数据，驱动下一代设计 | "传感器老在什么条件下出问题？要不要换规格？" |
| **售后技术支持** | 客户反馈问题后做远程初步诊断 | "不用去现场，先告诉我大概率是什么坏了" |

## 3. 为什么需要 AI 而不是再加一个看板

这是本 PRD 最核心的设计前提。

### 3.1 看板做不到的三件事

**A. 连接结构化数据和非结构化信息**

看板只能呈现"已经变成数字的东西"。制造厂大量有价值的信息是文本形态：

- 维修工单："更换左前激光雷达总成，密封圈有裂纹"
- 设计变更记录："Q4 起电机轴承供应商从 A 切换到 B，成本优化"
- 客户反馈："最近早上出车有异响，跑一会就好"
- 固件 Release Note："v3.2 优化了潮湿环境下的摄像头曝光参数"

当工程师问"SD22C-003 能耗为什么高了"，AI 不需要预先定义"查维修记录"这条规则——它能自己去翻这些文本，找到"003 上个月换过电机控制器，更换后能耗就上来了"，再往前翻"而且它是 Q4 批次的，Q4 正好换了轴承供应商"。

**B. 对话式探索，而非预设分析路径**

工程师排查问题不是按菜单走的。他的思路是：发现异常 → 追问一个方向 → 被结果引导到另一个方向 → 突然想起一个线索 → 再回头验证。每一步都依赖上一步的结果。看板的 drill-down 路径是写死的；对话是开放的。

**C. 用常识弥补数据盲区**

- "7 月深圳白天平均 35°C"——不在你的数据库里
- "高温会导致电池效率下降"——通用工程知识，不需要你编码为告警规则
- "沙尘环境会加速激光雷达密封圈老化"——物理常识

这些常识让 AI 在数据不全时仍然能给出有意义的推测方向——而非报一个"数据不足无法分析"。

### 3.2 AI 和看板的分工

| | 看板 | AI Agent |
|---|---|---|
| 回答"今天跑了多少公里" | ✅ | ✅（但不该用 AI 做这个） |
| 回答"能耗为什么高了" | ❌ | ✅ |
| 发现"补给节奏差的车辆利用率低 22%" | ❌ | ✅（主动关联发现） |
| 发现"更换轴承供应商后这批车能耗都涨了" | ❌ | ✅（跨数据源） |
| 回答"该不该召回这批车" | ❌ | 部分（AI 出分析，人做决策） |

**原则：能用看板回答的问题，不走 Agent。Agent 只做看板做不了的事。**

# 第二部分：数据架构

## 4. 数据源分层

Agent 需要连接的数据源，按接入优先级分三层：

### 4.1 L1 数据（必接——来自现有看板，结构化）

| 数据类别 | 具体内容 | 粒度 |
|---------|---------|------|
| 车辆运营指标 | 里程、能耗、在线率、完成率 | 单车/日 |
| 接管事件 | 时间、VIN、原因大类（行人横穿/标识误识别/施工绕行/道路封闭/其他） | 事件级 |
| 补给记录 | 充电/加水/倒垃圾的时间戳、VIN | 事件级 |
| 能耗构成 | 驱动/空调/附件三类占比 | 单车/日 |
| 告警记录 | 传感器故障/电量危险/通信异常等 | 事件级 |
| 车型基础信息 | 车型（SD15/SD22C）、累计行驶里程 | 单车 |

### 4.2 L2 数据（制造厂内部系统——关键差异化数据）

| 数据类别 | 具体内容 | 来源 |
|---------|---------|------|
| 生产批次 | 每辆车的生产批次/出厂日期/核心部件供应商 | MES/ERP |
| 固件版本 | 每辆车当前的固件/软件版本 + 历史更新记录 | OTA 平台 |
| 维修工单 | 维修时间、VIN、故障描述、更换部件、维修结论 | 售后系统 |
| 设计变更记录 | 变更时间、涉及部件、变更原因 | PLM/ECN |
| 客户反馈 | 客户报修/投诉的原始文本 | 客服/CRM |

### 4.3 L3 数据（远期——深化分析能力）

| 数据类别 | 具体内容 | 价值 |
|---------|---------|------|
| 车辆 CAN 总线数据 | 电机温度/转速、电池 SOC/温度/电压、制动回收效率 | 部件级健康检测 |
| 环境数据 | 部署城市的气温/湿度/降雨/沙尘 | 环境适应性分析 |
| 产线数据 | 每辆车的工位/拧紧力矩/检测值 | 一致性分析 |

## 5. 知识库（制造厂需要喂给 AI 的领域知识）

这是 AI 做因果推断的基础。不喂，AI 只能用常识推理；喂了，AI 才能做精准诊断。

| 知识类型 | 内容 | 来源 |
|---------|------|------|
| 故障模式库 | 已知故障模式 × 症状 × 根因 × 修复方案 | 工程师经验沉淀 |
| 部件规格 | 各部件的设计规格、寿命预期、失效模式 | 研发文档 |
| 供应商对照 | 各批次/部件对应的供应商及变更历史 | 采购记录 |
| 设计常识 | 车型设计决策及 trade-off（如"SD22C 电池容量比 SD15 大但电机功率也高"） | 研发文档 |
| 已知问题 | 已确认的设计缺陷、正在改进的项、临时 workaround | 质量看板 |

# 第三部分：Agent 核心能力

## 6. 能力 1：异源信息关联——把数字和文本串起来

**场景示例**：

> 工程师：SD22C-018 最近接管率从 1.2 涨到了 2.8，帮我查查。

Agent 的执行路径：

```
Step 1：拉取定量数据
├─ 接管率趋势：4 周前 1.2 → 3 周前 1.5 → 2 周前 1.9 → 本周 2.8
├─ 接管原因分布变化：激光雷达相关从 25% → 62%（+37pp）
├─ 其他指标：能耗 +3%、无显著变化
└─ 结论：问题集中在激光雷达系统

Step 2：搜索非结构化文本
├─ 维修记录中搜索 VIN=018 + 关键词"激光雷达"/"传感器"
│   └─ 命中：5 周前维修工单"018 报激光雷达自清洁异常，清洁后恢复"
├─ 设计变更中搜索"激光雷达"相关变更
│   └─ 命中：Q4 批次起激光雷达密封供应商从 A→B
├─ 确认 018 是否属于 Q4 批次
│   └─ 是，出厂日期 2025-11
└─ 搜索同批次其他车辆是否有类似趋势
    └─ 同批次 20 台中 6 台出现了类似接管率上升，7 台暂未出现但里程较低

Step 3：输出分析
├─ 018 接管率上升的主因是激光雷达系统性能退化
├─ 符合 Q4 批次的批量模式——激光雷达密封供应商变更后密封性下降
├─ 当前 6/20 台已出现症状，其余车辆可能随里程增加陆续出现
└─ 建议：
    1. 对 018 更换激光雷达总成
    2. 对 Q4 批次剩余 14 台做预防性检测
    3. 评估是否需要将密封供应商切回 A 或改进 B 的规格
```

**关键设计原则**：
- AI 自动关联文本搜索，不需要工程师手动翻工单/变更记录
- AI 自动扩展搜索范围——发现一个可疑方向（激光雷达密封），主动查同批次
- AI 给出的是"证据链 + 推测方向"，不是最终判决

## 7. 能力 2：对话式多轮探索——跟随工程师的思维

看板的 drill-down 是树状的，每层只能选预设分支。对话式探索是网状的——工程师在任何一步都能改变方向。

**场景示例**：

```
工程师：SD22C 最近整体接管率怎么样？

Agent：  近 4 周百公里接管率：1.2 → 1.3 → 1.5 → 1.6，趋势上升。
         TOP3 接管原因：激光雷达相关 35%、行为决策 22%、定位 15%。
         其中激光雷达相关接管涨幅最大（+4pp）。

工程师：按批次拆开看看。

Agent：  [批次对比表]
         Q3 批次（供应商 A）：接管率 1.2，稳定
         Q4 批次（供应商 B）：接管率 1.8，且每周 +0.2pp
         差异集中在激光雷达相关接管（Q4=48% vs Q3=21%）
         → Q4 批次激光雷达系统有系统性差异

工程师：Q4 批次还有哪些异常？

Agent：  扫描 Q4 批次 20 台车的全部指标：
         · 接管率：1.8（Q3=1.2）⚠
         · 百公里能耗：30.8（Q3=29.1）⚠
         · 补给节奏 CV：0.31（Q3=0.18）⚠
         · 在线率：93%（Q3=95%）基本持平
         → Q4 批次三个指标同步偏差，且一致性（标准差）均差于 Q3

工程师：把维修记录里涉及 Q4 批次激光雷达的都找出来。

Agent：  [Q4 批次 20 台车中已有 8 条激光雷达相关维修记录]
         故障描述高频词：密封圈、脏污、清洁失效、自清洁次数增多
         最早出现里程中位数：2100 km
         维修方式：更换总成 5 台 / 清洁复位 3 台
         清洁复位的 3 台中，2 台在 4-6 周后复发
```

每一步的追问都依赖上一步的发现。工程师不是在走预设菜单，而是**跟随证据链**。

## 8. 能力 3：主动巡检——发现没人想到要问的事

工程师不需要每天手动问 AI。Agent 按周期做自动扫描，发现值得关注的信号后主动报告。

### 8.1 巡检逻辑

```
每周自动执行一次全量车辆扫描：

1. 趋势检测
   ├─ 对每台车 × 每个核心指标，计算 4 周趋势斜率
   ├─ 标记 R² > 0.7 且方向为恶化（能耗升/接管升/在线率降）的 "趋势性异常"
   └─ 与单纯的 "阈值超标" 区分——还没超标但正在恶化的也要报告

2. 群体模式
   ├─ 按批次/车型/部署地区分组
   ├─ 对每组，检查是否有指标的系统性偏移
   └─ 标记 "组内均值偏离全局均值 > 1.5σ" 的组

3. 文本信号
   ├─ 扫描最新维修工单和客户反馈
   ├─ 提取高频关键词（部件名 + 故障描述）
   ├─ 与运营数据交叉验证
   └─ 标记 "文本中高频出现 + 运营数据中体现" 的共识信号

4. 新发异常
   ├─ 对比本周和上周各指标
   ├─ 标记突然恶化的车辆/批次
   └─ 排除已标注为 "已知问题" 的项（避免重复报告）
```

### 8.2 巡检报告格式

不生成冗长的列表，而是只列 **Top-N 最值得关注的事项**，每条包含：

- 严重程度（红/黄/蓝）
- 涉及范围（几台车 / 哪个批次 / 哪个部件）
- 证据（数字 + 文本交叉验证）
- 是否匹配已有故障模式（匹配则引用 / 不匹配则标注为"新信号"）

> **示例输出：本周值得关注（3 项）**
>
> 🔴 **Q4 批次激光雷达接管恶化** · 6 台已显症 · 匹配已知模式 #LDR-004
> Q4 批次 20 台中有 6 台激光雷达相关接管率 4 周内涨幅 > 100%。维修记录中 8 条激光雷达工单均指向密封圈问题。建议推进供应商纠正 + 预防性保养计划。
>
> 🟡 **SD15 电机温度趋势性上升** · 3 台 · 新信号（未匹配已知模式）
> VHC-02/07/11 三台车驱动电机峰值温度 30 天内趋势斜率 +0.3°C/天，R²>0.85。尚未超阈值，但趋势明确。建议安排电机检测并关注同批次车辆。
>
> 🔵 **客户反馈中"异响"出现频率上升** · 本月 6 次 vs 月均 2 次
> 6 条客户反馈共涉及 4 台车，均为累计里程 3000-5000km 的 SD22C。运营数据中暂未发现对应异常指标。建议跟踪并获取现场录音。

## 9. 能力 4：远程诊断——第一层技术支持

客户报修后，技术支持人员在去现场之前，先通过 Agent 做一遍远程诊断。

### 9.1 诊断流程

```
输入：客户公司名 + VIN + 问题描述（文本）

Step 1：快速画像
├─ 该车基本信息（车型/批次/累计里程/固件版本）
├─ 近 30 天核心指标趋势（里程/能耗/接管/在线）
└─ 与同车型均值的偏离

Step 2：问题定位
├─ 根据客户描述的问题类型，下钻到对应指标
│   ├─ "接管多" → 接管原因分布 + 按小时/按路线的聚集性
│   ├─ "能耗高" → 能耗构成拆解 + 补给节奏 CV
│   ├─ "跑不动" → 在线率 + 补给记录 + 完成率
│   └─ "故障灯亮" → 告警记录 + 故障码
├─ 搜索历史维修记录，检查是否有同类问题
├─ 检查固件版本是否为最新（不是 → 可能已通过 OTA 修复）
└─ 检查同批次/同公司其他车辆是否有类似表现

Step 3：诊断结论
├─ 判断问题范围（单车/批次/环境）
├─ 给出最可能的根因方向
├─ 给出建议处理方案
│   ├─ 远程可处理（推送固件/远程复位）
│   ├─ 需现场处理 + 建议更换部件清单
│   └─ 需升级到质量工程师（疑似批次问题）
└─ 引用支撑证据
```

### 9.2 诊断输出示例

> **远程诊断报告** · VHC-07（SD22C·Q4批次）
>
> **问题确认**：客户反馈"最近总接管"，数据确认百公里接管率 4 周内 0.8→2.4，属实。
>
> **根因推断**：激光雷达接管占比 60%，自清洁次数从 3次/天 增至 12次/天。匹配 Q4 批次已知密封圈问题（模式 #LDR-004）。同批次已有 6 台出现相同症状。
>
> **问题类型**：批次性问题（非单车偶发）
>
> **建议处理**：
> 1. 远程推送激光雷达自清洁参数重置（低风险，即刻执行）
> 2. 若 3 天内接管率未回落，安排现场更换激光雷达总成
> 3. 主动联系同批次另外 14 台车的客户，预约预防性检测
>
> **备注**：此车最新固件 v3.2，该问题与固件版本无关。

## 10. 能力 5：自由问答——兜底的自然语言查询

不是所有问题都是排查或诊断。工程师可能随时想问一个简单的事。

| 类型 | 示例 |
|------|------|
| 单指标查询 | "SD15 上个月总里程多少？" |
| 排序/筛选 | "能耗最高的 5 台车是谁？" |
| 分组聚合 | "按批次分组，对比 SD22C 的百公里能耗" |
| 同环比 | "Q4 批次这个月跟上个月接管率差多少？" |
| 条件筛选 | "累计里程超过 3000km 的 SD22C，有多少台有过传感器告警？" |
| 组合查询 | "Q3 批次、部署在南方的 SD15，哪台接管率最低？" |

**原则**：L1 数据范围内的任何单次查询都应该能直接回答。超出当前数据覆盖范围时明确告知边界。

# 第四部分：交互设计

## 11. 调用方式

- **部署位置**：公司级 Agent 平台（手写 Skill，手动调用）
- **调用方式**：@无人车分析 + 问题
- **首次使用**：Agent 自动告知当前可用数据范围和能力边界

## 12. 回答规范

### 12.1 分层呈现

| 层级 | 样式 | 用途 |
|------|------|------|
| **结论摘要** | 标题 + 粗体，回答开头 | 一句话说清楚"发现了什么" |
| **证据链** | 表格/列表/图表 | 数据的逐步下钻过程 |
| **建议/下一步** | 编号列表，回答结尾 | 可操作的下一步 |

### 12.2 数据透明

- 每条结论引数据来源（"VHC-07 近 4 周接管率 0.8 → 2.4" 而非 "接管率升高"）
- 数据不支持某个方向时明确说"当前数据无法判断"
- 区分"事实"和"推测"——后者用"推测"/"可能"/"建议确认"措辞

### 12.3 不确定性表达

| 置信度 | 表述 |
|--------|------|
| 匹配已知故障模式 | "符合已知模式 #LDR-004" |
| 多证据支持但未确证 | "高度怀疑，建议安排检测确认" |
| 单一数据线索 | "线索指向 X，但当前证据不足" |
| 未匹配已知模式 | "新信号，未匹配现有故障模式库" |

### 12.4 输出形式

支持富文本回答：结构化表格（对比/排名/趋势）、ECharts 图表（折线/柱状/散点）、卡片/标签（严重程度/状态）、纯文本（推测/建议）。

# 第五部分：边界与约束

## 13. Agent 不做的事

| 不做的事 | 原因 |
|---------|------|
| 最终根因判定 | 需工程师现场确认 |
| 召回/停运决策 | 需人做风险判断 |
| 维修操作指令 | 安全操作需人工审核 |
| 重复看板已有的统计查询 | 浪费 AI 资源 |
| 无数据支撑时编造结论 | 基本底线 |

## 14. 冷启动策略

- Phase 1：工程师在排查中将确认后的根因反馈给 Agent（手动标注），存入故障模式库
- Phase 2：案例积累足够后，Agent 可自动提议新故障模式（"最近 3 个月出现 5 次相似模式，建议建立新故障模式"）
- 冷启动期（前 3 个月）：允许 Agent 在"未匹配已知模式"时给出基于常识的推测，同时鼓励工程师反馈修正

# 第七部分：成功衡量

## 18. 关键指标

| 指标 | 衡量方式 | 目标 |
|------|---------|------|
| 使用频率 | 每周主动调用次数 | MVP 后 ≥ 15 次/周 |
| 诊断效率 | 远程诊断出结论的平均耗时 | < 3 分钟（vs 当前人工约 15-20 分钟） |
| 诊断质量 | 远程诊断后现场确认的吻合率 | ≥ 70%（一期允许推测性结论被修正） |
| 知识沉淀 | 工程师反馈建立的故障模式数量 | P1 结束时 ≥ 30 条 |
| 未覆盖率 | 提问中因数据/能力不足无法回答的比例 | < 30% |

## 19. 验收标准

**GIVEN** 工程师输入一条车辆异常的排查问题
**WHEN** Agent 分析了运营数据 + 搜索了维修工单和批次信息
**THEN** Agent 输出一条包含"数据趋势 + 文本关联 + 推测方向 + 建议下一步"的结构化回答，且每个推测标注了不确定性等级。
