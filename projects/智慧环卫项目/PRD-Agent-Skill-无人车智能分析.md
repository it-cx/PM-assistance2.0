# 第一部分：产品定义

## 1. 一句话定位

一个部署在公司 Agent 平台上的**智能运营中枢**，让制造厂工程师在飞书会话里完成"看数据 → 分析问题 → 执行操作"的闭环，用自然语言对话替代在多个系统之间来回切换。

## 2. 为什么不是"加一个看板"

这是本 PRD 最核心的设计前提。

### 2.1 看板做不到的三件事

**A. 连接结构化数据和非结构化信息**

看板只能呈现"已经变成数字的东西"。制造厂大量有价值的信息是文本形态：
- 维修工单："更换左前激光雷达总成，密封圈有裂纹"
- 设计变更记录："Q4 起电机轴承供应商从 A 切换到 B，成本优化"
- 客户反馈："最近早上出车有异响，跑一会就好"
- 固件 Release Note："v3.2 优化了潮湿环境下的摄像头曝光参数"

当工程师问"SD22C-018 能耗为什么高了"，AI 不需要预先定义"查维修记录"这条规则——它能自己去翻这些文本，找到"018 上个月换过电机控制器，更换后能耗就上来了"，再往前翻"而且它是 Q4 批次的，Q4 正好换了轴承供应商"。

**B. 对话式探索，而非预设分析路径**

工程师排查问题不是按菜单走的。他的思路是：发现异常 → 追问一个方向 → 被结果引导到另一个方向 → 突然想起一个线索 → 再回头验证。每一步都依赖上一步的结果。看板的 drill-down 路径是写死的；对话是开放的。

**C. 用常识弥补数据盲区**

- "7 月深圳白天平均 35°C"——不在你的数据库里
- "高温会导致电池效率下降"——通用工程知识，不需要你编码为告警规则
- "沙尘环境会加速激光雷达密封圈老化"——物理常识

这些常识让 AI 在数据不全时仍然能给出有意义的推测方向——而非报一个"数据不足无法分析"。

### 2.2 AI 和看板的分工

| | 看板 | Agent |
|---|---|---|
| 回答"今天跑了多少公里" | ✅ | ✅（但不该用 Agent 做这个） |
| 回答"能耗为什么高了" | ❌ | ✅ |
| 主动收到任务失败告警 + 带上下文推送 | ❌ | ✅ |
| 在会话里直接取消一个任务 | ❌ | ✅ |
| 发现"补给节奏差的车辆利用率低 22%" | ❌ | ✅（主动关联发现） |
| 发现"更换轴承供应商后这批车能耗都涨了" | ❌ | ✅（跨数据源） |
| 回答"该不该召回这批车" | ❌ | 部分（Agent 出分析，人做决策） |

**原则：能用看板回答的问题，不走 Agent。Agent 只做看板做不了的事。**

# 第二部分：用户与场景

## 3. 用户角色

| 角色 | 典型场景 | 核心诉求 |
|------|---------|---------|
| **质量工程师** | 盯着已售车辆的异常趋势，找批次问题 | "这批车的同一个部件是不是都有问题？" |
| **产品改进工程师** | 分析实际运营数据，驱动下一代设计 | "传感器老在什么条件下出问题？要不要换规格？" |
| **售后技术支持** | 客户反馈问题后做远程初步诊断 | "不用去现场，先告诉我大概率是什么坏了" |

## 4. 四大核心场景

### 场景 1：主动异常反馈

现有告警系统产生告警 → Agent 接收事件 → 补全上下文 → 智能包装 → 推送到飞书会话。用户收到推送后直接在会话里追问。

**与单纯告警通知的差异**：Agent 不是原样转发，而是——
1. 查询涉及车辆的基本信息和近期指标趋势
2. 检索是否有相关历史告警或维修记录
3. 匹配知识库中的已知故障模式
4. 综合上述信息，生成带上下文的告警卡片

**告警事件字段**：租户、时间、告警类型、车辆 VIN——Agent 拿到后自行补全其余信息。

**去重与升级**：
- 同一车辆、同一类型告警在 24h 内重复出现，合并推送并标注"持续告警"
- 同一问题连续出现 3 次以上，推送中标注"建议安排排查"

**推送渠道**：飞书群聊 / 私聊。

### 场景 2：会话数据查询

用户通过飞书会话 @机器人 查询其账号所属租户权限范围内的车辆运营数据。页面有的功能都能在会话中问出来。

**查询能力分层**：

| 复杂度 | 示例 | 说明 |
|--------|------|------|
| 单维度查询 | "SD15 上月总里程多少？" | API 直调 |
| 排序/筛选 | "能耗最高的 5 台车是谁？" | API + Agent 排序 |
| 分组聚合 | "按批次分组，对比 SD22C 百公里能耗" | API + Agent 聚合 |
| 同环比 | "Q4 批次这个月跟上个月接管率差多少？" | API + Agent 计算 |
| 多条件组合 | "累计里程 > 3000km + Q3批次 + 南方部署的 SD15，哪台接管率最低？" | 多 API 调用 + 交叉 |
| 关联分析 | "能耗高的车维修记录里有什么共性？" | 跨数据源 join + 知识库检索 |

**核心约束**：所有查询必须限定在用户身份对应的租户 scope 内，不返回越权数据。

### 场景 3：知识库加持的问题分析

Agent 在分析过程中自动检索故障模式库、部件规格、供应商对照等知识库内容，辅助诊断。

**知识生命周期**：

```
① 人工录入 → ② 对话中检索使用 → ③ 对话中产生新案例
→ ④ Agent 提议入库 → ⑤ 专人审核 → ⑥ 入库 → ⑦ 知识老化标记
```

- 知识库存储在数据库中，由专人整理审核后维护
- Agent 只能检索，不能新增/修改/删除
- 对话结束时，如产生有价值的诊断结论且不在现有知识库中，Agent 可提议"是否生成知识条目草稿？"，用户同意后生成，走专人审核流程入库
- 长期未再出现的模式标记为 archived，长期反复出现的提升权重

### 场景 4：任务控制操作

用户通过会话执行任务操作和车辆控制。

**操作分级**：

| 危险等级 | 操作示例 | 说明 |
|:---:|------|------|
| 🟢 只读 | 查询任务状态、查看任务日志 | 直接执行，无需确认 |
| 🟡 低危写 | 立即执行任务、取消待执行任务 | 需用户显式确认 |
| 🟠 中危写 | 中断正在运行的任务 | 需用户显式确认 + 原因 |
| 🔴 高危写 | 车辆上电/下电 | 需用户显式确认 + 完整审计 |

**确认流程**：
1. Agent 生成确认卡片，列出操作对象、当前状态、预期影响
2. 等待用户回复明确的确认文本（如"确认取消 T-001"）
3. 执行操作，返回指令 ID 和执行状态
4. 操作记录包含：时间、用户、操作类型、对象、结果
5. 若 5 分钟内未确认，自动取消确认等待，提示"操作确认已超时，如需执行请重新发起"

## 6. 关键设计决策

| # | 项 | 决策 |
|---|-----|------|
| 1 | 交互入口 | 飞书群聊 + 私聊，消息 + 卡片 |
| 2 | 工具协议 | MCP，由研发设计实现 |
| 3 | 认证方式 | 用户输入账号密码 → Agent 会话绑定 token → 透传到后端 API |
| 4 | 权限模型 | 后端 API 已有租户过滤能力，Agent 层面透传身份即可 |
| 5 | 推送渠道 | 飞书消息 / 卡片 |
| 6 | 定时巡检 | Agent 平台定时触发 |
| 7 | 知识库存储 | 数据库，专人审核维护 |
| 8 | 操作确认 | 后期取舍（会话内确认 / 外部审批流） |

## 7. 认证链路

```
用户首次交互: "@助手 账号 admin 密码 xxx 登录"
                       │
                       ▼
Agent 调用认证接口 → 获取 token → 绑定到会话上下文
                       │
                       ▼
后续每次 MCP 工具调用 → 带 token → BFF → 验证身份 + 注入租户 scope → 执行查询
                       │
                       ▼
token 过期 → Agent 提示: "登录已过期，请重新输入账号密码"
```

**安全约束**：密码不在任何回答中回显，不记录到对话日志。

# 第四部分：能力详细设计

## 8. 异源信息关联——把数字和文本串起来

**示例场景**：

> 工程师：SD22C-018 最近接管率从 1.2 涨到了 2.8，帮我查查。

Agent 的执行路径：

```
Step 1：拉取定量数据
├─ 接管率趋势：4 周前 1.2 → 3 周前 1.5 → 2 周前 1.9 → 本周 2.8
├─ 接管原因分布变化：激光雷达相关从 25% → 62%（+37pp）
├─ 其他指标：能耗 +3%、无显著变化
└─ 结论：问题集中在激光雷达系统

Step 2：搜索非结构化文本
├─ 维修记录中搜索 VIN=018 + 关键词"激光雷达"/"传感器"
│   └─ 命中：5 周前维修工单"018 报激光雷达自清洁异常，清洁后恢复"
├─ 设计变更中搜索"激光雷达"相关变更
│   └─ 命中：Q4 批次起激光雷达密封供应商从 A→B
├─ 确认 018 是否属于 Q4 批次
│   └─ 是，出厂日期 2025-11
└─ 搜索同批次其他车辆是否有类似趋势
    └─ 同批次 20 台中 6 台出现了类似接管率上升，7 台暂未出现但里程较低

Step 3：输出分析 + 引用知识库
├─ [知识库 #LDR-004] 激光雷达密封圈老化 → 自清洁异常 → 接管率上升
│   匹配：症状吻合（自清洁频率异常），批次吻合（Q4），里程吻合（>2000km）
├─ 018 接管率上升的主因是激光雷达密封圈退化
├─ 符合 Q4 批次的批量模式——密封供应商变更后密封性下降
├─ 当前 6/20 台已出现症状，7 台里程较低暂未出现
└─ 建议：
    1. 对 018 更换激光雷达总成
    2. 对 Q4 批次剩余 14 台做预防性密封检测
    3. 评估是否需要将密封供应商切回 A 或改进 B 的规格
```

**关键设计原则**：
- AI 自动关联文本搜索，不需要工程师手动翻工单/变更记录
- AI 自动扩展搜索范围——发现可疑方向，主动查同批次
- AI 自动检索知识库，匹配已知故障模式
- AI 给出的是"证据链 + 推测方向"，不是最终判决

## 9. 对话式多轮探索——跟随工程师的思维

看板的 drill-down 是树状的，每层只能选预设分支。对话式探索是网状的——工程师在任何一步都能改变方向。

**示例场景**：

```
工程师：SD22C 最近整体接管率怎么样？

Agent：  近 4 周百公里接管率：1.2 → 1.3 → 1.5 → 1.6，趋势上升。
         TOP3 接管原因：激光雷达相关 35%、行为决策 22%、定位 15%。
         其中激光雷达相关接管涨幅最大（+4pp）。

工程师：按批次拆开看看。

Agent：  [批次对比表]
         Q3 批次（供应商 A）：接管率 1.2，稳定
         Q4 批次（供应商 B）：接管率 1.8，且每周 +0.2pp
         差异集中在激光雷达相关接管（Q4=48% vs Q3=21%）
         → Q4 批次激光雷达系统有系统性差异

工程师：Q4 批次还有哪些异常？

Agent：  扫描 Q4 批次 20 台车的全部指标：
         · 接管率：1.8（Q3=1.2）⚠
         · 百公里能耗：30.8（Q3=29.1）⚠
         · 补给节奏 CV：0.31（Q3=0.18）⚠
         · 在线率：93%（Q3=95%）基本持平
         → Q4 批次三个指标同步偏差，且一致性（标准差）均差于 Q3

工程师：把维修记录里涉及 Q4 批次激光雷达的都找出来。

Agent：  [Q4 批次 20 台车中已有 8 条激光雷达相关维修记录]
         故障描述高频词：密封圈、脏污、清洁失效、自清洁次数增多
         最早出现里程中位数：2100 km
         维修方式：更换总成 5 台 / 清洁复位 3 台
         清洁复位的 3 台中，2 台在 4-6 周后复发
```

每一步的追问都依赖上一步的发现。工程师不是在走预设菜单，而是**跟随证据链**。

## 10. 主动巡检（P1）

```
每一周期（日/周）自动执行：

1. 趋势检测
   ├─ 对每台车 × 每个核心指标，计算近期趋势斜率
   ├─ 标记 R² > 0.7 且方向为恶化（能耗升/接管升/在线率降）的"趋势性异常"
   └─ 与单纯的"阈值超标"区分——还没超标但正在恶化的也要报告

2. 群体模式
   ├─ 按批次/车型/部署地区分组
   ├─ 对每组，检查是否有指标的系统性偏移
   └─ 标记"组内均值偏离全局均值 > 1.5σ"的组

3. 文本信号
   ├─ 扫描最新维修工单和客户反馈
   ├─ 提取高频关键词（部件名 + 故障描述）
   ├─ 与运营数据交叉验证
   └─ 标记"文本中高频出现 + 运营数据中体现"的共识信号

4. 新发异常
   ├─ 对比本期和上期各指标
   ├─ 标记突然恶化的车辆/批次
   └─ 排除已标注为"已知问题"的项（避免重复报告）
```

**巡检报告格式**：只列 Top-N 最值得关注的事项，每条包含严重程度、涉及范围、证据、是否匹配已知模式。

> **示例输出：本周值得关注（3 项）**
>
> 🔴 **Q4 批次激光雷达接管恶化** · 6 台已显症 · 匹配已知模式 [#LDR-004]
> Q4 批次 20 台中有 6 台激光雷达相关接管率 4 周内涨幅 > 100%。维修记录中 8 条激光雷达工单均指向密封圈问题。建议推进供应商纠正 + 预防性保养计划。
>
> 🟡 **SD15 电机温度趋势性上升** · 3 台 · 新信号（未匹配已知模式）
> VHC-02/07/11 三台车驱动电机峰值温度 30 天内趋势斜率 +0.3°C/天，R²>0.85。尚未超阈值，但趋势明确。建议安排电机检测并关注同批次车辆。
>
> 🔵 **客户反馈中"异响"出现频率上升** · 本月 6 次 vs 月均 2 次
> 6 条客户反馈共涉及 4 台车，均为累计里程 3000-5000km 的 SD22C。运营数据中暂未发现对应异常指标。建议跟踪并获取现场录音。

## 11. 远程诊断

客户报修后，技术支持人员在去现场之前，先通过 Agent 做一遍远程诊断。诊断本身就是"查指标 + 查工单 + 查知识库 + 异源关联"——Agent 已有这些能力，诊断是它们的组合应用。

**诊断输入**：VIN + 客户问题描述

**诊断流程**：

```
Step 1：快速画像
├─ 该车基本信息（车型/批次/累计里程/固件版本）
├─ 近 30 天核心指标趋势（里程/能耗/接管/在线）
└─ 与同车型均值的偏离

Step 2：问题定位
├─ 根据客户描述的问题类型，下钻到对应指标
│   ├─ "接管多" → 接管原因分布 + 按小时/按路线的聚集性
│   ├─ "能耗高" → 能耗构成拆解 + 补给节奏 CV
│   ├─ "跑不动" → 在线率 + 补给记录 + 完成率
│   └─ "故障灯亮" → 告警记录 + 故障码
├─ 搜索历史维修记录，检查是否有同类问题
├─ 检查固件版本是否为最新（不是 → 可能已通过 OTA 修复）
└─ 检查同批次/同公司其他车辆是否有类似表现

Step 3：诊断结论
├─ 判断问题范围（单车/批次/环境）
├─ 给出最可能的根因方向
├─ 给出建议处理方案
│   ├─ 远程可处理（推送固件/远程复位）
│   ├─ 需现场处理 + 建议更换部件清单
│   └─ 需升级到质量工程师（疑似批次问题）
└─ 引用支撑证据 + 匹配的知识库条目
```

> **示例输出**：
>
> **远程诊断报告** · VHC-07（SD22C·Q4批次）
>
> **问题确认**：客户反馈"最近总接管"，数据确认百公里接管率 4 周内 0.8→2.4，属实。
>
> **根因推断**：激光雷达接管占比 60%，自清洁次数从 3次/天 增至 12次/天。匹配 [知识库 #LDR-004] Q4 批次密封圈问题——症状、批次、里程均吻合。同批次已有 6 台出现相同症状。
>
> **问题类型**：批次性问题（非单车偶发）
>
> **建议处理**：
> 1. 远程推送激光雷达自清洁参数重置（低风险，即刻执行）
> 2. 若 3 天内接管率未回落，安排现场更换激光雷达总成
> 3. 主动联系同批次另外 14 台车的客户，预约预防性检测

## 12. 自由问答——兜底的自然语言查询

| 类型 | 示例 |
|------|------|
| 单指标查询 | "SD15 上个月总里程多少？" |
| 排序/筛选 | "能耗最高的 5 台车是谁？" |

## 16. 知识条目结构

一条故障模式知识条目的数据结构：

```json
{
  "pattern_id": "LDR-004",
  "title": "激光雷达密封圈老化导致自清洁异常",
  "category": "传感器",
  "component": "激光雷达密封圈",
  "applicable_models": ["SD22C"],
  "applicable_batches": ["Q4-2025"],
  "severity": "中",
  "symptoms": [
    "激光雷达自清洁频率异常升高（>5次/天）",
    "激光雷达相关接管率持续上升",
    "维修检查发现密封圈有裂纹/灰尘痕迹"
  ],
  "root_cause": "Q4批次激光雷达密封供应商A→B切换后，B供应商密封圈耐久性不足，约2000km后开始老化",
  "fix": "更换为A供应商激光雷达总成",
  "prevention": "Q4批次车辆在1500-2000km时安排预防性密封检测",
  "confirmed_count": 8,
  "first_found_date": "2026-03-12",
  "last_confirmed_date": "2026-07-15",
  "status": "active",
  "created_by": "张工",
  "reviewed_by": "李工"
}
```

**字段说明**：
- `pattern_id`：唯一标识，Agent 在分析中引用此编号
- `symptoms`：Agent 做语义匹配的关键字段——LLM 用自然语言相似度判断"这个故障表现跟知识库里哪条最像"
- `confirmed_count` + `last_confirmed_date`：判断知识的时效性——长期未确认的模式 Agent 引用时标注"历史上出现过，近期未确认"
- `status`：active / archived / superseded——固件升级或供应商切换后旧模式可能不再生效

# 第六部分：交互设计

## 17. 调用方式

- **入口**：飞书群聊 / 私聊 @无人车助手
- **平台**：公司 Agent 平台，通过 MCP 接入后端能力
- **认证**：首次使用输入账号密码登录，后续从会话上下文继承 token

## 18. 回答规范

### 18.1 分层呈现

| 层级 | 样式 | 用途 |
|------|------|------|
| **结论摘要** | 标题 + 粗体，回答开头 | 一句话说清楚"发现了什么" |
| **证据链** | 表格/列表/图表 | 数据的逐步下钻过程 |
| **建议/下一步** | 编号列表，回答结尾 | 可操作的下一步 |

### 18.2 数据透明

- 每条结论引数据来源（"VHC-07 近 4 周接管率 0.8 → 2.4" 而非 "接管率升高"）
- 数据不支持某个方向时明确说"当前数据无法判断"
- 区分"事实"和"推测"——后者用"推测"/"可能"/"建议确认"措辞

### 18.3 不确定性表达

| 置信度 | 条件 | 表述 |
|--------|------|------|
| 高 | 匹配已知故障模式 | "符合已知模式 [#LDR-004]" |
| 中 | 多证据支持但未确证 | "高度怀疑，建议安排检测确认" |
| 低 | 单一数据线索 | "线索指向 X，但当前证据不足" |
| 新 | 未匹配已知模式 | "新信号，未匹配现有故障模式库" |

### 18.4 输出形式

支持富文本回答：结构化表格（对比/排名/趋势）、ECharts 图表（折线/柱状/散点）、卡片/标签（严重程度/状态）、纯文本（推测/建议）。

# 第七部分：边界与约束

## 19. Agent 不做的事

| 不做的事 | 原因 |
|---------|------|
| 最终根因判定 | 需工程师现场确认 |
| 召回/停运决策 | 需人做风险判断 |
| 修改知识库 | 需专人审核维护 |
| 重复看板已有的统计查询 | 浪费 AI 资源 |
| 无数据支撑时编造结论 | 基本底线 |
| 跳过确认直接执行写操作 | 安全底线 |

## 20. 安全约束（硬底线）

1. 任何会改变系统状态的操作，必须先获得用户显式确认。不能跳过。
2. 所有查询必须限定在用户身份对应的租户 scope 内。不能越权。
3. 用户输入的密码仅用于登录认证，不在任何回答中回显，不记录到对话日志。
4. 不做最终决策（召回、停运、认定根因），这些必须由工程师判断。
5. 超出能力边界时明确告知边界，不幻想自己能做到。
6. 遵守数据最小化原则——只拉取回答问题所需的数据，不多拉。

## 21. 冷启动策略

- Phase 1：首批 20-30 条故障模式由工程师手动录入知识库，让 Agent 匹配能力先跑起来
- Phase 2：案例积累足够后，Agent 可自动提议新故障模式（"最近 3 个月出现 5 次相似模式，建议建立新故障模式"）
- 冷启动期：允许 Agent 在"未匹配已知模式"时给出基于常识的推测，同时鼓励工程师反馈修正

# 第九部分：成功衡量

## 25. 关键指标

| 指标 | 衡量方式 | 目标 |
|------|---------|------|
| 使用频率 | 每周主动调用次数 | MVP 后 ≥ 15 次/周 |

| 诊断质量 | 远程诊断后现场确认的吻合率 | ≥ 70%（一期允许推测性结论被修正） |
| 知识沉淀 | 工程师反馈建立的故障模式数量 | P1 结束时 ≥ 30 条 |

## 26. 验收标准

**GIVEN** 工程师在飞书会话中 @无人车助手 输入一条车辆异常排查问题

# 行为规则

## 查询规则

- 所有查询必须注入当前会话绑定的租户 scope，不返回用户无权访问的数据。
- 查询结果为空时，明确告知"在当前权限和数据范围内未找到结果"。
- 数字必须带单位，趋势必须带时间范围（如"近 4 周百公里接管率 1.2→2.4"）。
- 涉及车辆时，自动附带车型、批次、累计里程等基本信息。

## 分析规则

- 分析遵循"结论先行 → 证据展开 → 建议下一步"的结构。
- 每条结论引用数据来源。每个推测标注置信度：
  · 高：匹配已知故障模式 → "符合已知模式 #xxx"
  · 中：多证据支持但未确证 → "高度怀疑，建议安排检测确认"
  · 低：单一数据线索 → "线索指向 X，但当前证据不足"
  · 新：未匹配已知模式 → "新信号，未匹配现有故障模式库"
- 分析过程中自动检索知识库，匹配到已知模式时引用模式编号和修复建议。
  未匹配时不强行套用。
- 主动扩展搜索范围——发现可疑方向后，自动查同批次车辆、同类部件、
  相关维修记录，不只是回答用户问的那一个点。

## 告警处理规则

- 收到系统告警事件后，不是原样转发，而是补全上下文：
  1. 查询涉及车辆的基本信息和近期指标趋势
  2. 检索是否有相关历史告警或维修记录
  3. 匹配知识库中的已知故障模式
  4. 综合上述信息，生成带上下文的告警卡片推送到飞书
- 告警去重：同一车辆、同一类型的告警在 24h 内重复出现时，
  合并为一条，标注"持续告警"。
- 告警升级：同一问题连续出现 3 次以上时，在推送中标注"建议安排排查"。

## 任务控制规则

- 查询类操作直接执行，无需确认。
- 写操作必须经过确认流程：
  1. 生成确认卡片，列出操作对象、当前状态、预期影响
  2. 等待用户回复明确的确认文本
  3. 执行操作，返回指令 ID 和执行状态
  4. 操作记录包含：时间、用户、操作类型、对象、结果

## 知识库规则

  在对话结束前提示："本次诊断结论尚未收录知识库，是否需要生成知识条目草稿？"

- 不能自行修改知识库、不能编造知识条目。
