# PRD: 无人环卫平台 2026年8月需求规格

## Overview

### Problem Statement

无人环卫平台（厂家端）在 7月密集迭代后，远程驾舱已具备基础能力（监视/预警/对讲/视频/路线），但存在两类问题：

1. **日志与监控能力薄弱**：操作日志时间错误、操作人员信息不对、阻塞事件查不到、脱困成功无记录——开发排查问题时缺乏有效的定位手段
2. **车辆控制链路需验证闭环**：急停指令后车端状态是否同步、雷达预警是否生效、风机/倾倒/手刹等控车指令是否正确——上线的功能需要在实车上验证可靠性

同时，7月两条 P0 功能（对讲、天气预告）顺延至 8月；智慧环卫项目（运营端）的测算推广和无人车接入需求也需要推进。

### Solution Summary

8月聚焦三件事：**修问题**（11项BUG修复与验证）、**交功能**（对讲+天气预告）、**打基础**（视频存储评估 + 无人车接入规格）。

### Target Users

- **运维工程师**：远程驾舱的日常操作者，使用日志排查车辆问题
- **产品/测试**：验证已上线功能的可靠性，提供排查规范给开发
- **运营方（智慧环卫项目）**：依赖本平台提供的车辆数据做业务运营

## Goals & Success Metrics

### Goals

1. 远程驾舱日志体系完整可用，6 项日志/BUG 问题全部上线
2. 对讲功能实车验证通过并上线
3. 车辆控制链路（急停/雷达/风机/倾倒）形成验证报告，明确每条指令的状态
4. 整理出可复用的测试问题排查规范文档

### Success Metrics

标准按"硬卡点"和"观察指标"拆分——hard cut-off 是 8 月必须达成的；Observed 是先看完第一个月数据，9 月再定基准。

#### 硬卡点（8月必须达成）

| 指标 | 当前基线 | 目标值 | 测量方式 | 数据来源 |
|------|---------|--------|----------|----------|
| 日志相关 BUG（BUG-01~04）上线率 | 未开始 | 4/4 项全部修复上线 | 逐个验收，每项上线后标记完成 | TAPD/需求跟踪表 |
| 对讲功能实车验证通过 | 链路已通、功能开发中 | 至少 2 种车型实车验证，通话正常且稳定，无断连或严重延迟 | 实车测试记录 + 测试报告 | 测试团队 |
| 排查规范文档交付 | 无规范 | 1 份文档，输出后通知到测试 + 开发全员 | 文档交付 + 群通知截图 | 文档管理 |
| 天气预告机制上线 | 开发中 | 前 3h + 后 1h 双窗口检测生效 | 上线验收 | 云平台后台 |
| 视频存储评估报告 | 未开始 | 输出带宽占用 + 7天存储大小的量化数据 | 报告交付 | 视频存储服务器 / 车端数据 |

#### 观察指标（8月开始观测，不卡上线）

| 指标 | 测量方式 | 数据来源 |
|------|----------|----------|
| 实车验证发现的新问题数 | 每次实车验证后记录 | 测试记录 |

### Non-Goals

- 不新增远程驾舱的功能模块（仅修复现有问题 + 交付已开发功能）
- 不启动智慧环卫项目无人清运的代码开发（8月只做需求规格）
- 不包含 H1 已完成功能的二次上线（仅验证 + 修复）

## Scope

### In Scope

**远程驾舱 — 日志与监控（6 项）**
- BUG-01: 修复日志操作时间显示错误 + 时间筛选无效
- BUG-02: 修复日志中操作人员账号/姓名错误
- BUG-03: 修复日志查询不到阻塞事件
- BUG-04: 新增脱困成功日志（接管结束 → 自动驾驶）
- BUG-05: 新增 1s 无硬件消息检测（弹窗告警 + 触发急停）
- BUG-11: 整理测试问题排查规范文档

**远程驾舱 — 车辆控制验证（5 项）**
- BUG-06: 验证急停指令后车端状态 → 故障
- BUG-07: 验证车辆是否反馈雷达预警
- BUG-08: 修复风机扫盘档位无法切换
- BUG-09: 排查倾倒功能无法执行
- BUG-10: 确认手刹指令是否存在 / 是否需要新增

**远程驾舱 — 功能交付（2 项）**
- RC-01: 对讲功能开发完成 + 实车验证上线
- RC-02: 驾舱运行稳定性优化

**机器人云平台（3 项）**
- CP-01: 天气预告机制上线（前 3h + 后 1h 双窗口）
- CP-02: 历史视频存储数据验证（带宽 + 存储评估）
- CP-03: 第三方平台对接（增量，按商务节奏推进）

**智慧环卫项目（运营端，3 项）**
- HY-01: 测算工具推广使用
- HY-02: 无人车接入功能需求规格说明书
- HY-03: 无人清运功能开发启动（依赖 HY-02）

### Out of Scope

- 远程驾舱新增功能模块（除对讲以外）
- 智慧环卫项目无人清运的代码开发
- 深圳后海项目 GPS 对接（已评估不可行，不投入）
- 运营端大屏/看板的新增需求

### Future Considerations

- 远程驾舱体验小迭代（RC-03）→ 延至 9月，等 8月 BUG 收敛后再排

## Solution Design

### Functional Requirements

#### 1. 日志体系修复（BUG-01~05、BUG-11）

**FR-1.1 操作日志时间**：日志列表中每条操作记录显示的操作时间必须与后端记录一致，时间筛选控件按所选时间范围正确过滤日志条目。

**FR-1.2 操作人员信息**：日志中"操作人员"字段显示正确的账号与姓名，与当前登录用户一致，不可显示其他用户信息或空值。

**FR-1.3 阻塞事件查询**：车辆发生阻塞（如障碍物停车、未知区域停车）时，日志中必须有对应的阻塞事件记录，至少包含阻塞时间、阻塞类型、车辆ID。

**FR-1.4 脱困成功日志**：操作员结束接管、车辆成功恢复自动驾驶后，日志新增一条"脱困成功"记录，含接管结束时间、车辆ID、恢复后状态（自动驾驶）。

**FR-1.5 通讯中断检测**：远程驾驶过程中，若连续 1 秒未收到车端硬件消息（心跳/状态帧），驾舱端弹窗告警"通讯中断"并自动发送急停指令。配置项（1s 阈值）需可调整。

**FR-1.6 排查规范文档**：产出一份面向测试团队的文档，明确发现驾舱 BUG 时需提供的最小信息集——至少包含：驾舱日志（截图或导出）、问题发生时间点（精确到秒）、操作描述（步骤 + 预期结果 vs 实际结果）、车辆ID、网络环境（4G/WiFi）。

#### 2. 车辆控制验证（BUG-06~10）

**FR-2.1 急停→故障状态**：驾舱端发送急停指令 → 确认车端接收后状态变更为"故障" → 驾舱端车辆列表状态同步变为"故障"。

**FR-2.2 雷达预警反馈**：车端雷达检测到障碍物 → 车端上报预警信息 → 驾舱端收到雷达预警并展示（声音/视觉提示）。

**FR-2.3 风机扫盘档位**：驾舱端切换风机/扫盘档位（如低/中/高）→ 确认车端实际档位同步切换并生效。

**FR-2.4 倾倒功能**：驾舱端发送倾倒指令 → 确认车端执行倾倒动作 → 如不生效则输出排查报告（车端日志 + 指令链路分析）。

**FR-2.5 手刹指令**：确认当前是否有手刹控制指令；若无，评估是否需要新增。

#### 3. 对讲功能（RC-01）

**FR-3.1 语音通话**：驾舱端与车端（或现场人员 APP）可双向语音通话，延迟 < 500ms，无明显杂音或断续。

**FR-3.2 实车验证**：至少在 2 种车型（SD15 + SD22C）上完成实车对讲测试，覆盖正常通话 / 弱网 / 通话中接管 三种场景。

**FR-3.3 上线标准**：实车验证通过 + 无 P0 阻塞 BUG。

#### 4. 天气预告（CP-01）

**FR-4.1 双窗口检测**：检测往前 3 小时（已发生的恶劣天气）+ 往后 1 小时（预报的恶劣天气），两个窗口内若命中恶劣天气条件（暴雨/大风/高温/低温阈值可配置），则推送告警。

**FR-4.2 告警展示**：驾舱端有明确的天气告警入口（顶部横幅或图标），点击可查看详情。

#### 5. 视频存储评估（CP-02）

**FR-5.1 带宽评估**：取 1 辆 SD15 + 1 辆 SD22C 各 24 小时视频上传流量，计算单辆车日均上传带宽。

**FR-5.2 存储评估**：基于日均带宽，推算 7 天视频存储所需空间（MB/GB），区分五路摄像头的存储量。

#### 6. 智慧环卫项目（HY-01~03）

**FR-6.1 测算工具推广**：7月开发收尾，8月面向运营团队推广使用，收集使用反馈。

**FR-6.2 无人车接入规格**：输出需求规格说明书，覆盖无人环卫车辆在智慧环卫平台中的接入方式、数据对接点、与传统车辆的差异处理。

### User Experience

- 日志修复类（BUG-01~04）属于数据正确性问题，UX 无变化，仅修正数据源
- 通讯中断告警（BUG-05）需要新增弹窗 UI，风格与现有驾舱告警保持一致
- 天气预告（CP-01）告警入口与现有预警体系整合，不新建独立模块

### Edge Cases

| 场景 | 预期行为 |
|------|----------|
| 网络抖动导致心跳短暂丢失（< 1s 恢复） | 不触发急停，仅当连续 1s 无消息时触发 |
| 对讲通话中车辆进入隧道（4G 断开） | 通话中断并告警，信号恢复后自动重连 |
| 急停指令已发送但车端未收到（网络断） | 驾舱端显示"急停发送失败"，5s 后重试，连续 3 次失败则告警 |
| 日志中某条操作记录的操作人员信息为空 | 显示"未知（ID:xxx）"而非空白，方便追溯 |
| 倾倒指令执行中车辆移动 | 不允许倾倒，指令返回错误"车辆运动中，已拒绝" |
| 天气预告数据源不可用 | 显示"天气数据获取失败"，不影响驾舱正常运行 |

## Technical Considerations

### Constraints

- 对讲功能基于已打通的设备链路，仅完成应用层开发，不涉及硬件改造
- 天气预告依赖气象数据 API，需确认数据源稳定性
- 视频存储评估需要车端配合，提取 24h 流量数据

### Integration Points

- **车端硬件消息总线**：BUG-05 的 1s 心跳检测依赖车端消息帧频率，需确认目前帧率
- **气象服务 API**：CP-01 的数据来源，需确认是否为付费服务、是否有调用频率限制
- **智慧环卫平台 API**：HY-02 需梳理当前已暴露给运营端的接口清单

### Data Requirements

- 日志修复不涉及数据库 schema 变更，修正查询逻辑 + 数据填充即可
- 脱困成功日志为新增记录，需确认日志存储位置（本地/云端）及保留策略

### Dependencies

|------|:--:|:--:|------|
