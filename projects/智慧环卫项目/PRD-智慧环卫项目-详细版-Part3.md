### 9.2 Hero 指标区（3 行 × 3 指标）

| 指标 | 格式 | 说明 |
|------|------|------|
| **总作业里程** | `X,XXX km` | 当天所有车辆行驶里程总和 |
| **总耗电量** | `XXX kWh` | 当天所有车辆耗电总和 |
| **总运行时长** | `XX.X h` | 当天所有车辆运行时间总和 |

**数据更新**：日期切换 → `getData(dim)` 根据日期种子生成当日数值（含±15% 随机抖动模拟真实波动）

### 9.3 Hero 趋势图（renderHeroTrend）

**图表类型**：折线 + 面积图，7 天/30 天切换

| 配置 | 值 |
|------|-----|
| X 轴 | 日期（MM-DD），category |
| Y 轴 | 里程（km） |
| 数据 | `dailyMileage` 数组，根据 `dim` 参数切换（7天=过去7天，30天=过去30天） |
| 折线 | 蓝色 `#0EA5E9`，smooth:true，宽度 2px |
| 面积 | 半透明渐变填充 `#0EA5E933` → transparent |
| 点标记 | 当天数据点突出显示（symbolSize:8） |
| 切换按钮 | 图表标题右侧 [7天] [30天] 分段按钮，切换后重新渲染 |
| 保护 | try-catch 包裹，ECharts 不可用时静默跳过 |

### 9.8 资源补给节奏模块（不受维度影响，24 小时视图）

**三行分类：**
1. 充电（⚡）
2. 加水（💧）
3. 倒垃圾（🗑）

**每行展示：**
- 24 小时时间轴横条
- 补给事件标记点（圆点或短竖线），位置 = 发生时刻
- 统计：补给次数、平均间隔（分钟）

**节奏效率评级（CV 变异系数）：**

```
CV = 间隔标准差 / 间隔均值
A 级: CV < 0.2  → 节奏稳定，站点调度合理
B 级: 0.2 ≤ CV ≤ 0.4 → 正常
C 级: CV > 0.4  → 补给忽早忽晚，调度有问题
```

**视觉呈现**：
- A 级 → 绿色标签
- B 级 → 黄色标签
- C 级 → 红色标签
- 每个补给事件 tooltip：显示具体时间（如 "14:32"）、间隔（如 "上间隔 1h23min"）

**异常聚集标记（P2 预留）**：后续检测连续多个补给事件间隔过短（排队拥堵）的情况

## 10. 能效弹窗详情（Modal）

### 10.1 触发方式
点击能效看板区域 → `openModal('efficiency')`

### 10.2 弹窗布局

```
┌────────────────────────────────────────────────┐
│  能效详情                           [✕]         │
├────────────────────────────────────────────────┤
│  总耗电量: 138 kWh                               │
├──────────────────┬─────────────────────────────┤
│  SD15            │  SD22C                       │
│  百km能耗: 22.5   │  百km能耗: 30.2              │
│  自驾占比: 87%    │  自驾占比: 82%               │
│  能耗构成(饼图)   │  能耗构成(饼图)               │
└──────────────────┴─────────────────────────────┘
```

**KPI 拆分原则：**
- **总耗电量**：不拆分（所有车型合计）
- **百公里能耗**：按车型独立计算
- **自驾占比**：按车型独立计算（自动驾驶里程 / 总里程）
- **能耗构成**：每个车型一张独立饼图（`effPieSD15` / `effPieSD22C`）

**数据来源**：
- `totalEnergy` = 所有车辆 energy 总和
- `modelKwh100` = 该车型 energy 总和 / 该车型 mileage 总和 × 100
- `autoDrivePct` = 该车型自动驾驶里程占比（当前 Mock 数据，后续接入实际）

点击安全告警区域 → `openModal('safety')`

## 11. 协同事件（2.3）

### 11.1 功能背景

SD22C 大型清扫车的充电、加水、排污清理目前需人工协作。需要在平台上建立缺水缺电告警 → 网格指派 → APP 通知的闭环。

### 11.2 事件流程

```
SD22C缺水/缺电 → 车端上报 → 平台接收
→ 查询加水点/充电点所在网格
→ 匹配网格负责人
→ 推送消息至环卫云APP
→ 负责人处理（完成/转派）
→ 事件关闭
```

### 11.3 功能字段

| ID | 功能 | 优先级 | 字段/逻辑 |
|------|------|:--:|------|
| F-J01 | 缺水缺电告警上报 | P0 | 车辆 VIN、告警类型（缺水/缺电/缺排污）、上报时间、当前位置（GPS）、当前电量/水量百分比 |
| F-J02 | 网格匹配 | P0 | 网格 ID（预定义网格边界）、网格负责人 ID + 姓名、匹配逻辑：根据车辆 GPS 反向地理编码 → 判断所在网格 → 查询该网格负责人 |
| F-J03 | APP 推送 | P0 | 通知渠道：环卫云 APP 推送（主）+ 短信（降级）；通知内容：车辆 VIN + 位置 + 告警类型 + 预计到达加水/充电点时间；确认方式：负责人点击"接受" |
| F-J04 | 事件追踪 | P1 | 事件状态：待响应 → 处理中 → 已完成 / 已转派；每个状态变更记录时间戳 + 操作人；超时升级：待响应超过 N 分钟 → 推送上级管理员 |
| F-J05 | 智能协作事件类型 | P1 | 在实时事件流（9.9）中新增"智能协作事件"类别，蓝色图标，与接管/告警/天气并列 |
