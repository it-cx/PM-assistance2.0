## 严重问题（阻断性，必须修复）

- [ ] **点位状态机缺失** — 位置：§3.1 收运点管理
  - PRD定义了点的属性（名称/经纬度/类型/策略），但未定义点的生命周期。点位从录入到废弃，中间可能经历什么状态？至少需要：启用、暂停、废弃。如果某个点位因施工暂时不可用，排班模板引用了这个点怎么办？
  - **建议**：增加点位的状态字段和流转规则——启用↔暂停↔废弃；排班模板生成任务时跳过"暂停/废弃"的点位并告警。

- [x] ~~**点位"负责人"与排班"跟车人员"概念混淆**~~ — **已修正 v1.1**
  - 用户确认：没有"跟车人员"角色。现场人员即点位负责人，一个人负责若干收运点，车快到通知对应点位负责人前往配合。已删除"跟车人员"角色，统一为"点位负责人"。

- [x] ~~**策略B的二次超时无兜底**~~ — **已修正 v1.1**
  - F-19 已增加：管理员超时未响应（默认 10 分钟，可配置）→ 自动降级为策略A（跳过+通知操作员）。

- [x] ~~**收运确认无撤销机制**~~ — **已修正 v1.1**
  - F-17 改为车端物理按键 + 长按 3 秒确认，物理操作天然防误触，无需撤销机制。

## 中等问题（影响体验或完整性，建议修复）

- [x] ~~**车辆故障/电量危险的中途处理未覆盖**~~ — **已确认，有远程监管人员处理**
  - 用户确认：车辆故障、电量危险等车辆端异常由远程监管人员负责处理。PRD 新增"远程监管人员"角色，该部分逻辑不在收运场景 PRD 范围内。

- [ ] **APP推送与短信的触发条件未区分** — 位置：§3.4 F-16
  - PRD 写"APP推送/短信"，但短信有成本（每条 0.03-0.05 元），对所有通知都发短信不经济。需要区分通知优先级。
  - **建议**：到站通知 → APP 推送优先；超时告警 → APP推送 + 短信双通道；管理员决策通知 → 短信。在NF需求中增加"短信月度成本预估"。

- [ ] **排班周期缺少"节假日例外"** — 位置：§3.3 F-12
  - F-12 支持每日/每周/自定义间隔，但没有"节假日跳过"或"节假日特殊班次"。如果客户春节不作业、国庆减少班次，调度员需要手动暂停模板。
  - **建议**：F-12 增加"节假日策略"配置——跳过/执行/自定义班次。支持导入法定节假日日历。

- [ ] **路线执行中无动态调整能力** — 位置：§3.2, §3.4
  - 路线一旦保存排班，执行中无法临时调整。实际运营中可能出现：某个点临时需要加收（垃圾桶溢出投诉）、某个点因施工封路需跳过。目前只能通过操作员手动干预，缺少系统化支持。
  - **建议**：增加"任务中动态调整"——操作员可对执行中的任务临时添加/跳过收运点，系统重新规划剩余路径，记录调整日志。

- [ ] **数据分析指标的"满载率"数据来源不明** — 位置：§3.6 F-28
  - "满载率"指标需要称重或体积传感器数据。如果车辆没有该传感器，"满载率"无法计算。
  - **建议**：明确满载率的数据源——是有传感器的精确值还是基于点位垃圾量预估值的估算值。如果是估算值，排除该指标或标注为"预估值"。

- [ ] **中转站容量和开放时间未作为约束** — 位置：§3.2 F-09
  - 路线绑定中转站，但中转站可能容量饱和或非24小时运营。如果车辆凌晨 4 点到中转站但站 8 点才开门，或站已满无法卸货，车辆白跑一趟。
  - **建议**：F-09 增加中转站开放时段和容量上限配置。排班时检查中转站在预计到达时间的可接纳状态。

- [ ] **清扫车与收运车在大屏上的共存规则缺失** — 位置：§3.5
  - PRD多次提到现有清扫车+新增收运车，但大屏需求未明确两者在大屏上的展示关系：同一地图图层？不同图标？能否一键切换车型视图？
  - **建议**：F-22 增加"车型筛选"——支持按清扫车/收运车切换视图；不同车型使用不同颜色图标；默认全显示。

## 轻微问题（改进建议，可选修复）

- [ ] **F-01 批量导入的格式未细化** — 位置：§3.1 F-01
  - 只说"支持批量导入 CSV/Excel"，未定义字段模板。运维人员拿到客户数据后不知道要填哪些列。
  - **建议**：提供"下载导入模板"功能，模板包含所有必填字段列，附带示例行。

- [ ] **策略分配逻辑可简化** — 位置：§3.1 F-03/F-04
  - 当前逻辑：重点标记→默认策略B+手动覆盖；普通→默认策略A+手动覆盖。实际运营中可能总是"重点=策略B、普通=策略A"，两层配置略显冗余。
  - **建议**：简化：点位只有一个属性"策略"（A/B），默认A。标记重点时提示"建议切换为策略B"。减少概念冗余。

- [ ] **验收标准缺少"月/周排班模板"场景** — 位置：§6
  - 验收标准只覆盖了"每日排班"，但平台支持每周和自定义间隔。缺少"每周一三五执行"这种场景的验收。
  - **建议**：补充一条：Given 每周一三五的排班模板 → When 启用模板 → Then 仅在周一/周三/周五自动生成任务、周二/周四不生成。

- [ ] **数据分析 F-31 垃圾量趋势依赖数据积累** — 位置：§3.6 F-31
  - PRD 已标注"需逐步积累数据"，但未给出冷启动方案——上线初期没有历史数据时，F-31 显示什么？
  - **建议**：冷启动期显示"数据积累中（预计XX天后可用）"，占位 UI 避免空白图表。同时用 F-10 的垃圾量预估作为初始参考值。

- [ ] **"车辆"概念在排班和路线中未明确类型约束** — 位置：§3.3 F-11
  - F-11 说"指定车辆（或车辆类型）"，但收运车可能有不同载重规格。如果路线预估垃圾量 8 吨，但调度员选了 3 吨的小车，排班时应给出建议。
  - **建议**：F-11 选车时展示车辆载重能力，与路线预估总量对比，若载重不足给出提示（不强制阻止，允许调度员自行判断）。

- [ ] **F-15 冲突检测范围偏窄** — 位置：§3.3 F-15
  - 当前仅检测"同一车辆被多重占用"，未检测"同一跟车人员被多重分配"或"同一收运点在同一时段被两条路线覆盖"。
  - **建议**：扩展冲突检测到人员和时间窗口重叠。

## 亮点（做得好的地方，继续保持）

- ✅ Grilling 阶段把收运场景的特殊性（点位级策略、人车协同、通知→响应→兜底）问得很透，没有生搬清扫车的逻辑
- ✅ 异常兜底设计现实——接受"人车协同"而非追求全自动，两种策略覆盖了不同重要等级的点位
- ✅ 排班采用模板化设计——避免重复操作，降低调度员日活成本
- ✅ 里程碑按功能模块递进（点→线→排班→执行→大屏），M6 集成测试作为最终关卡，顺序合理
- ✅ 竞品分析作为 PRD 前置输入，让收运蓝海定位有数据支撑
- ✅ 待确认项诚实标注，不自己假设 APP 复用和技术细节
